# Deploying NSX-Translated Policies on AWS OpenShift

**Date**: 2026-07-07
**Version**: 1.0

---

## Overview

The MultiNetworkPolicies generated by ops-translate from NSX workflows **CAN be deployed on AWS-based OpenShift clusters**, but require adaptation because AWS doesn't support traditional 802.1Q VLAN tagging.

**What translates well**:
- ✅ MultiNetworkPolicy manifests (no changes needed)
- ✅ Network segmentation concepts (VLANs → VPC subnets)
- ✅ Policy enforcement on secondary networks

**What needs adaptation**:
- ⚠️ NetworkAttachmentDefinitions (change CNI from macvlan/VLAN to ipvlan or macvlan without VLAN)
- ⚠️ Infrastructure (Multiple ENIs instead of VLAN interfaces)

---

## AWS Networking Options for Secondary Networks

### Option 1: Multiple ENIs (Recommended for AWS)

Use AWS Elastic Network Interfaces to provide multiple network attachments to pods.

**Architecture**:
```
NSX VLAN 100 (10.10.100.0/24)  →  AWS VPC Subnet A (10.10.100.0/24)
NSX VLAN 150 (10.10.150.0/24)  →  AWS VPC Subnet B (10.10.150.0/24)
NSX VLAN 200 (10.10.200.0/24)  →  AWS VPC Subnet C (10.10.200.0/24)
```

**Prerequisites**:
1. Create VPC subnets matching NSX subnet CIDRs
2. Attach multiple ENIs to EC2 worker nodes
3. Configure NetworkAttachmentDefinitions to use those ENIs

**Example NAD for AWS**:
```yaml
apiVersion: k8s.cni.cncf.io/v1
kind: NetworkAttachmentDefinition
metadata:
  name: web-tier-subnet-a
  namespace: virt-lab
spec:
  config: |-
    {
      "cniVersion": "0.3.1",
      "type": "ipvlan",
      "master": "eth1",              # Secondary ENI on EC2 instance
      "mode": "l2",
      "ipam": {
        "type": "whereabouts",
        "range": "10.10.100.0/24",   # Same subnet as NSX VLAN 100
        "range_start": "10.10.100.10",
        "range_end": "10.10.100.250",
        "gateway": "10.10.100.1",
        "routes": [{"dst": "0.0.0.0/0"}]
      }
    }
```

**Key Differences from On-Prem**:
- `"type": "ipvlan"` instead of `"type": "macvlan"`
- No `"vlan"` field (not supported on AWS)
- `"master": "eth1"` refers to secondary ENI, not VLAN interface

### Option 2: AWS VPC CNI with Secondary IPs

Use AWS VPC CNI plugin to assign secondary IP addresses from VPC subnets.

**Architecture**:
- Pods get secondary IPs from VPC subnets
- No additional ENIs needed
- Limited to IPs in node's subnet

**Example NAD**:
```yaml
apiVersion: k8s.cni.cncf.io/v1
kind: NetworkAttachmentDefinition
metadata:
  name: web-tier-vpc-subnet
  namespace: virt-lab
spec:
  config: |-
    {
      "cniVersion": "0.3.1",
      "type": "aws-cni",
      "subnetId": "subnet-0123456789abcdef0",  # AWS subnet ID
      "securityGroups": ["sg-0123456789abcdef0"]
    }
```

**Limitations**:
- Requires AWS VPC CNI plugin
- IP allocation limited by ENI limits
- Less isolation than separate ENIs

### Option 3: SR-IOV (High Performance)

Use SR-IOV for high-performance workloads requiring dedicated network hardware.

**Prerequisites**:
- EC2 instance types that support SR-IOV (e.g., i3.metal, m5.metal)
- OpenShift SR-IOV Network Operator installed

**Example NAD**:
```yaml
apiVersion: k8s.cni.cncf.io/v1
kind: NetworkAttachmentDefinition
metadata:
  name: web-tier-sriov
  namespace: virt-lab
spec:
  config: |-
    {
      "cniVersion": "0.3.1",
      "type": "sriov",
      "vfio": true,
      "ipam": {
        "type": "whereabouts",
        "range": "10.10.100.0/24"
      }
    }
```

**Limitations**:
- Expensive instance types
- Limited number of VFs per instance
- More complex setup

## Step-by-Step: Deploy on AWS OpenShift

### Prerequisites

1. **OpenShift 4.12+ on AWS**
2. **Cluster admin access**
3. **VPC with multiple subnets** matching NSX subnet CIDRs
4. **EC2 instances with multiple ENIs** (for Option 1)

### Step 1: Enable MultiNetworkPolicy

Same as on-premises:

```bash
# Check if enabled
oc get network.operator.openshift.io cluster -o jsonpath='{.spec.useMultiNetworkPolicy}'

# Enable if needed
oc patch network.operator.openshift.io cluster --type=merge \
  -p '{"spec":{"useMultiNetworkPolicy":true}}'

# Wait for network operator
oc wait --for=condition=Progressing=False --timeout=300s co/network
```

### Step 2: Configure AWS Infrastructure

#### Create VPC Subnets

Create subnets matching your NSX segments:

```bash
# Web tier subnet (was VLAN 100)
aws ec2 create-subnet \
  --vpc-id vpc-0123456789abcdef0 \
  --cidr-block 10.10.100.0/24 \
  --availability-zone us-east-1a \
  --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=web-tier-subnet}]'

# App tier subnet (was VLAN 150)
aws ec2 create-subnet \
  --vpc-id vpc-0123456789abcdef0 \
  --cidr-block 10.10.150.0/24 \
  --availability-zone us-east-1a \
  --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=app-tier-subnet}]'

# DB tier subnet (was VLAN 200)
aws ec2 create-subnet \
  --vpc-id vpc-0123456789abcdef0 \
  --cidr-block 10.10.200.0/24 \
  --availability-zone us-east-1a \
  --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=db-tier-subnet}]'
```

#### Attach Additional ENIs to Worker Nodes

For each OpenShift worker node:

```bash
# Get worker instance ID
INSTANCE_ID=$(oc get nodes -l node-role.kubernetes.io/worker -o jsonpath='{.items[0].metadata.name}' | xargs -I {} aws ec2 describe-instances --filters "Name=private-dns-name,Values={}" --query 'Reservations[0].Instances[0].InstanceId' --output text)

# Create and attach ENI for web tier
WEB_ENI=$(aws ec2 create-network-interface \
  --subnet-id subnet-web-tier-id \
  --description "Web tier secondary network" \
  --query 'NetworkInterface.NetworkInterfaceId' \
  --output text)

aws ec2 attach-network-interface \
  --network-interface-id $WEB_ENI \
  --instance-id $INSTANCE_ID \
  --device-index 1

# Repeat for app tier and db tier subnets...
```

### Step 3: Adapt Generated NetworkAttachmentDefinitions

Replace the generated NADs with AWS-compatible versions:

```bash
# Original (on-prem with VLAN)
cat output/network-attachments/web-tier-vlan100.yaml

# Becomes (AWS with ipvlan)
cat > output/network-attachments/web-tier-vlan100-aws.yaml <<'EOF'
apiVersion: k8s.cni.cncf.io/v1
kind: NetworkAttachmentDefinition
metadata:
  name: web-tier-vlan100
  namespace: virt-lab
  labels:
    translated-from: nsx-segment
    source-workflow: nsx-3tier-app.workflow
    vlan-id: "100"
    aws-subnet: "web-tier-subnet"
spec:
  config: |-
    {
      "cniVersion": "0.3.1",
      "type": "ipvlan",
      "master": "eth1",           # Secondary ENI (not VLAN interface)
      "mode": "l2",
      "ipam": {
        "type": "whereabouts",
        "range": "10.10.100.0/24",
        "range_start": "10.10.100.10",
        "range_end": "10.10.100.250",
        "gateway": "10.10.100.1",
        "routes": [{"dst": "0.0.0.0/0"}]
      }
    }
EOF
```

**Key changes**:
1. `"type": "macvlan"` → `"type": "ipvlan"`
2. Remove `"vlan": 100` field
3. `"master": "eth1"` references secondary ENI
4. Keep subnet information from NSX analysis

### Step 4: Deploy Resources

```bash
# Deploy adapted NADs
oc apply -f output/network-attachments/web-tier-vlan100-aws.yaml -n virt-lab
oc apply -f output/network-attachments/app-tier-vlan150-aws.yaml -n virt-lab
oc apply -f output/network-attachments/db-tier-vlan200-aws.yaml -n virt-lab

# Deploy MultiNetworkPolicies (NO CHANGES NEEDED!)
oc apply -f output/multi-network-policies/ -n virt-lab

# Deploy standard NetworkPolicies
oc apply -f output/network-policies/ -n virt-lab
```

**Important**: MultiNetworkPolicies generated by ops-translate work **unchanged** on AWS!

### Step 5: Verify Setup

```bash
# Check NADs
oc get network-attachment-definitions -n virt-lab

# Check policies
oc get multi-networkpolicy.k8s.cni.cncf.io -n virt-lab
oc get networkpolicies -n virt-lab
```

### Step 6: Test with Pods

```bash
# Create test pod
cat > test-pod-aws.yaml <<'EOF'
apiVersion: v1
kind: Pod
metadata:
  name: web-server
  namespace: virt-lab
  labels:
    app: websecuritygroup
  annotations:
    k8s.v1.cni.cncf.io/networks: web-tier-vlan100
spec:
  containers:
  - name: nginx
    image: nginxinc/nginx-unprivileged:alpine
EOF

oc apply -f test-pod-aws.yaml

# Verify secondary interface
oc exec -n virt-lab web-server -- ip addr show
# Should see: eth0 (primary), net1 (secondary from web-tier subnet)
```

## Mapping NSX Concepts to AWS

| NSX Concept | AWS Equivalent | Implementation |
|-------------|----------------|----------------|
| VLAN Segment | VPC Subnet | Create subnet with same CIDR |
| VLAN ID (100, 150, 200) | Subnet ID | Use subnet tags for mapping |
| VLAN Interface | Secondary ENI | Attach ENI in target subnet |
| NSX Distributed Firewall | MultiNetworkPolicy | Deploy unchanged |
| Security Group | AWS Security Group | Apply to ENIs (optional) |
| NSX Gateway | VPC Router | Default VPC routing |

## Automation Script

Here's a script to automate NAD conversion for AWS:

```bash
#!/bin/bash
# convert-nads-for-aws.sh

# Convert generated NADs from VLAN/macvlan to ipvlan for AWS

for nad_file in output/network-attachments/*.yaml; do
  filename=$(basename "$nad_file")

  # Skip README
  if [[ "$filename" == "README.md" ]]; then
    continue
  fi

  # Extract VLAN ID from label
  vlan_id=$(grep 'vlan-id:' "$nad_file" | awk '{print $2}' | tr -d '"')

  # Convert NAD
  cat "$nad_file" | \
    sed 's/"type": "macvlan"/"type": "ipvlan"/' | \
    sed '/"vlan":/d' | \
    sed 's/"mode": "bridge"/"mode": "l2"/' | \
    sed 's/TODO: Specify parent interface (e.g., eth1, ens3)/eth1  # Secondary ENI/' \
    > "${nad_file%.yaml}-aws.yaml"

  echo "Converted $filename -> ${filename%.yaml}-aws.yaml"
done

echo ""
echo "✅ Conversion complete!"
echo ""
echo "Next steps:"
echo "1. Review *-aws.yaml files"
echo "2. Verify 'eth1' matches your secondary ENI device name"
echo "3. Apply to cluster: oc apply -f output/network-attachments/*-aws.yaml"
```

**Usage**:
```bash
chmod +x convert-nads-for-aws.sh
./convert-nads-for-aws.sh
```

## Limitations on AWS

### What Works
- ✅ MultiNetworkPolicy enforcement (unchanged)
- ✅ Secondary network isolation via VPC subnets
- ✅ Pod-to-pod communication within secondary networks
- ✅ IP address management with Whereabouts

### What Doesn't Work
- ❌ Traditional 802.1Q VLAN tagging
- ❌ VLAN trunking across physical switches
- ❌ Direct hardware VLAN interfaces

### Performance Considerations
- **ipvlan**: Lower overhead than macvlan, good performance
- **Multiple ENIs**: Limited by EC2 instance type (e.g., m5.large supports 3 ENIs)
- **IP limits**: Each ENI has max IPs based on instance type
- **SR-IOV**: Best performance but requires metal instances

---

## Cost Implications

### Additional AWS Costs
1. **Additional ENIs**: $0.005/hour per ENI (varies by region)
2. **Elastic IPs**: If using public IPs on secondary networks
3. **Data transfer**: Inter-subnet traffic within same AZ is free
4. **NAT Gateway**: If secondary networks need internet access

### Cost Optimization
- Use single ENI with secondary IPs when possible
- Keep secondary networks private (no internet gateway)
- Use VPC endpoints for AWS service access
- Leverage same availability zone for all subnets

Let's deploy the NSX 3-tier app on AWS OpenShift:

### 1. Create VPC Infrastructure
```bash
# Create VPC subnets matching NSX segments
```

```bash
./convert-nads-for-aws.sh
```


## Troubleshooting

### Pod fails with "no such device eth1"

**Cause**: Secondary ENI not attached or different device name

**Fix**:
```bash
# Check available interfaces on node
oc debug node/<node-name> -- chroot /host ip link show

# Update NAD with correct device name
oc edit network-attachment-definition web-tier-vlan100 -n virt-lab
```

### IP allocation fails

**Cause**: Whereabouts IPAM not installed or IP range exhausted

**Fix**:
```bash
# Install Whereabouts
oc apply -f https://raw.githubusercontent.com/k8snetworkplumbingwg/whereabouts/master/doc/daemonset-install.yaml

# Or increase IP range in NAD
```

### MultiNetworkPolicy not enforcing

**Cause**: Feature not enabled

**Fix**:
```bash
oc patch network.operator.openshift.io cluster --type=merge \
  -p '{"spec":{"useMultiNetworkPolicy":true}}'
```
